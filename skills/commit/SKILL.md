# commit

## 目标
产出易于审查且可安全发布的提交：
- 默认将整个工作区按逻辑拆成多条小提交，直到工作区干净
- 每条提交的边界清晰，必要时主动拆分
- 提交信息准确说明改了什么以及为什么改
- 默认使用中文编写提交信息
- 提交风格：必须使用 Conventional Commits。

## 默认规则
- 默认目标是处理整个工作区，而不是只处理“这次请求范围”。
- 默认连续创建多条提交，直到 `git status --short` 为空。
- 默认提交信息使用中文；只有在用户明确要求英文，或仓库规则明确要求英文时，才改用英文。
- 如果用户明确说“只提交我这次改动”“不要动其他改动”“保留某些改动不提交”，再切换为范围受限模式。

## 工作流
1) 先检查整个工作区
   - `git status --short`
   - 如果改动很多：`git diff --stat`
   - `git diff`（未暂存改动）
   - `git diff --cached`（已暂存改动）
   - 先理解工作区里有哪些独立变更，再决定需要拆成几条提交。
2) 决定下一条最小合理提交
   - 拆分维度包括：功能与重构、后端与前端、格式化与逻辑、测试与生产代码、依赖升级与行为变化。
   - 优先把最独立、最容易描述清楚的一组改动单独拿出来先提交。
   - 如果一个文件里混有多类改动，优先按文件精确暂存；只有在确实需要时才使用补丁暂存。
   - 如果你无法用 1 到 2 句话说清“改了什么 / 为什么改”，通常说明这条提交仍然太大。
3) 只暂存属于下一条提交的内容
   - 默认优先使用非交互式命令精确暂存：`git add <path>`
   - 取消暂存优先使用非交互式命令：`git restore --staged <path>`
   - 只有在必须做 hunk 级拆分、且当前环境适合交互式 git 时，再使用 `git add -p` 或 `git restore --staged -p`
4) 审查实际将被提交的内容
   - `git diff --cached`
   - 基本检查项：
     - 没有密钥或令牌
     - 没有意外遗留的调试日志
     - 没有无关的格式化噪音
     - 这条提交的主题单一，和其他待提交改动边界清楚
5) 先用 1 到 2 句话描述已暂存改动
   - 回答两个问题：“改了什么？”和“为什么改？”
   - 这两句话应能直接指导提交标题和正文。
6) 编写提交信息
   - 必须使用 Conventional Commits：
     - `type(scope): 中文摘要`
     - 空行
     - 正文可选；需要时写中文的 what/why，不写实现流水账
     - 如有需要，补充 footer（`BREAKING CHANGE`）
   - 默认使用中文；只有用户或仓库明确要求英文时才切换。
   - 非交互式提交优先使用 `git commit -m "type(scope): 中文摘要"`；多段信息可使用多个 `-m`
   - 如果当前环境适合交互式编辑，再使用 `git commit -v`
7) 运行最小但相关的验证
   - 在提交前运行仓库里最快且有意义的检查，例如单元测试、lint 或构建。
8) 提交后回到第 1 步
   - 再次检查工作区，继续处理下一条提交。
   - 默认重复这个循环，直到工作区干净。
   - 只有当用户明确限制提交范围时，才提前停止。

## 提交信息模板
### 默认语言
- 默认使用中文。
- 只有在用户明确要求英文，或仓库规则明确要求英文时，才切换为英文。

### 简短模板
适用于已暂存改动足够小、标题本身就能说清楚的情况：

```text
type(scope): 中文摘要
```

示例：

```text
docs(commit): 调整默认提交流程
```

### 带正文模板
适用于审查者还需要更多上下文的情况：

```text
type(scope): 中文摘要

改了什么，为什么要改。
聚焦行为、意图和审查上下文。
不要写成实现流水账。
```

示例：

```text
docs(commit): 默认连续提交直到工作区干净

将默认目标改为处理整个工作区，而不是只处理当前请求范围。
保留例外门：如果用户明确限定范围，再切换为范围受限模式。
```

### Type 提示
- `feat`: 用户可见的新能力或行为变化
- `fix`: 缺陷修复或错误行为修正
- `docs`: 纯文档改动
- `refactor`: 不改变外部行为的内部调整
- `test`: 新增或调整测试
- `chore`: 维护性或工具性改动

### 快速检查
- 标题清楚说明这条提交改了什么。
- 正文解释为什么这条改动值得提交。
- 提交信息描述的是已暂存 diff，而不是整个分支。
- `scope` 是可选的；如果没有增益，可以省略。
- 默认使用中文，除非用户或仓库另有要求。

## 交付物
需要提供：
- 每条提交的最终提交信息
- 每条提交的简要说明（改了什么 / 为什么改）
- 用于检查、暂存和审查的命令（至少包含 `git diff --cached`，以及实际运行过的测试）
- 是否已处理到工作区干净；如果没有，说明停止原因
